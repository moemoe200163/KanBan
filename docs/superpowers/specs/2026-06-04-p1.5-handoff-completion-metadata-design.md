# P1.5 — Handoff Completion Metadata Design

**Date:** 2026-06-04
**Sprint:** P1.5
**Scope:** Data contract sprint. No UI refactor.
**Status:** Approved (Approach A)

---

## 1. Goal

Promote Handoff completion payloads from a freeform `dict` to a lane-specific
strongly-typed Pydantic schema. Each of the 8 worker lanes (`triage`, `product`,
`architect`, `frontend`, `backend`, `qa`, `review`, `delivery`) declares its
required completion fields with explicit types, length bounds, and enum
constraints. The backend rejects invalid completions with a structured 422
response; the existing frontend completion form surfaces the field-level errors
with a minimal parsing tweak.

This is a **data contract sprint**, not a UI rebuild. The user-visible form
keeps its current shape and behavior.

## 2. Non-Goals

The following are explicitly out of scope for P1.5:

- **OpenAPI surface change** (Approach B): payload remains `Optional[dict]` at
  the HTTP boundary. Pydantic validation happens in the service layer.
- **Codegen to frontend** (Approach C): no `pydantic-to-typescript` /
  `datamodel-code-generator`. Frontend types stay loose.
- **Strong typing at the `dispatch()` endpoint**: only `approver` presence is
  required at dispatch time. Lane-specific schema applies to `complete()`.
- **Cross-field validation** (e.g. `screenshots` paths must exist on disk).
- **Workflow automation** (`decision == "approve"` auto-advances the lane).
- **Frontend per-field input types** (number / select / textarea): the form
  stays text-input. Only the error parsing path is updated to read
  `err.data?.detail`.
- **Backfilling legacy payloads** to the new contract: existing handoffs are
  not re-validated retroactively.
- **Audit event payload typing**: `IssueEvent.payload` stays as freeform JSON.
- **Dynamic lane CRUD / admin lane editor**: deferred per existing
  `CLAUDE.md` non-goals.

## 3. Architecture

### 3.1 New module — `backend/core/kanban_protocol/payloads.py`

Single source of truth for per-lane completion contracts.

```python
"""Per-lane completion payload schemas.

Each worker lane declares the shape of the payload required to mark a handoff
complete. The validation happens in HandoffService.complete() — not at the
HTTP boundary — so the API surface stays stable while the contract becomes
strict.
"""
from typing import Literal
from pydantic import BaseModel, ConfigDict, Field, ValidationError


class CompletionPayloadBase(BaseModel):
    """All completion payloads inherit this base.

    `extra="forbid"` is **defense-in-depth**: it stops unknown fields from
    reaching the DB. The scope guard (`check_payload`) is the primary line
    of defense for the out-of-scope blacklist (e.g. `sandbox_egress`,
    `iptables_rules`). Both layers must be present.
    """
    model_config = ConfigDict(extra="forbid", str_strip_whitespace=True)


class TriagePayload(CompletionPayloadBase):
    lane_recommendation: Literal[
        "frontend", "backend", "qa", "architect", "product"
    ]
    summary: str = Field(..., min_length=1, max_length=2000)


class ProductPayload(CompletionPayloadBase):
    acceptance_criteria: list[str] = Field(..., min_length=1)


class ArchitectPayload(CompletionPayloadBase):
    design_notes: str = Field(..., min_length=1, max_length=10000)
    interfaces: list[str] = Field(..., min_length=1)


class FrontendPayload(CompletionPayloadBase):
    diff_summary: str = Field(..., min_length=1, max_length=4000)
    screenshots: list[str] = Field(default_factory=list)


class BackendPayload(CompletionPayloadBase):
    diff_summary: str = Field(..., min_length=1, max_length=4000)
    test_results: str = Field(..., min_length=1, max_length=4000)


class QaPayload(CompletionPayloadBase):
    test_results: str = Field(..., min_length=1, max_length=4000)
    coverage_pct: int = Field(..., ge=0, le=100)


class ReviewPayload(CompletionPayloadBase):
    reviewer: str = Field(..., min_length=1, max_length=128)
    decision: Literal["approve", "reject", "request_changes"]


class DeliveryPayload(CompletionPayloadBase):
    release_notes: str = Field(..., min_length=1, max_length=20000)


LANE_PAYLOADS: dict[str, type[BaseModel]] = {
    "triage": TriagePayload,
    "product": ProductPayload,
    "architect": ArchitectPayload,
    "frontend": FrontendPayload,
    "backend": BackendPayload,
    "qa": QaPayload,
    "review": ReviewPayload,
    "delivery": DeliveryPayload,
}


class PayloadValidationError(ValueError):
    """Raised by HandoffService.complete() when a lane's payload fails
    Pydantic validation.

    Carries the structured Pydantic errors so the API layer can produce a
    machine-readable 422 response.
    """
    def __init__(self, lane: str, errors: list[dict]) -> None:
        self.lane = lane
        self.errors = errors
        super().__init__(
            f"Payload validation failed for lane '{lane}': "
            f"{len(errors)} error(s)"
        )
```

### 3.2 `lanes.py` derives `required_completion_fields`

`WorkerLane.required_completion_fields` becomes a property, derived from
`LANE_PAYLOADS[self.key].model_fields.keys()`. This keeps the human-readable
list in sync with the schema. Existing readers (preview endpoint, frontend
preview) need no call-site changes.

```python
@property
def required_completion_fields(self) -> List[str]:
    from core.kanban_protocol.payloads import LANE_PAYLOADS
    return list(LANE_PAYLOADS[self.key].model_fields.keys())
```

### 3.3 Guardrails (Hard Requirements)

Three guardrails bind this design. They are pinned by tests.

1. **`check_payload(final_payload)` runs before Pydantic validation.**
   If it didn't, Pydantic's `extra="forbid"` would silently drop out-of-scope
   keys before `check_payload` could see them, defeating the scope guard.
2. **All schemas inherit `CompletionPayloadBase` with `extra="forbid"`.**
   Unknown fields raise `ValidationError` (`type="extra_forbidden"`).
3. **Frontend surfaces structured 422 errors.** Without this, the typed
   contract's value is invisible to the user. One targeted parsing tweak
   in `HandoffSection.vue::submitCompletion` is allowed (not a UI refactor).

## 4. Service-Layer Integration

`HandoffService.complete()` in `backend/core/kanban_protocol/handoff.py`
applies the contract in this order:

```python
async def complete(
    self, *, handoff_id: str, actor: Optional[str], payload: Optional[dict]
) -> dict:
    current = await repo.get_issue_handoff(handoff_id)
    if not current:
        raise ValueError(f"Handoff '{handoff_id}' not found")
    if current["status"] not in ("in_progress", "accepted"):
        raise ValueError(
            f"Cannot complete handoff in status '{current['status']}'; "
            "only 'in_progress' or 'accepted' handoffs can be completed"
        )

    lane = get_lane(current["toLane"])
    existing = current.get("payload") or {}
    caller = payload or {}
    final_payload = {**existing, **caller}

    # Guardrail 1: scope guard runs FIRST.
    check_payload(final_payload)

    # Then lane-specific Pydantic validation.
    payload_model = LANE_PAYLOADS[lane.key]
    try:
        validated = payload_model.model_validate(final_payload)
    except ValidationError as exc:
        raise PayloadValidationError(
            lane=lane.key,
            errors=[
                {"loc": list(e["loc"]), "msg": e["msg"], "type": e["type"]}
                for e in exc.errors()
            ],
        ) from exc

    merged_payload = validated.model_dump(mode="json")

    return await repo.update_issue_handoff(
        handoff_id,
        status="completed",
        payload=merged_payload,
        actor_field="completed_by",
        actor_value=actor,
        set_completed_at=True,
    )
```

`check_payload` (scope guard) is unchanged; it lives in
`core/kanban_protocol/scope_guard.py` and remains the authority on which keys
are off-limits for security reasons.

## 5. API Layer — Structured 422

`complete_handoff` in `backend/api/v1/endpoints/handoffs.py` adds a new
exception branch:

```python
@router.post(
    "/boards/{board_id}/issues/{issue_id}/handoffs/{handoff_id}/complete"
)
async def complete_handoff(
    board_id: str, issue_id: str, handoff_id: str,
    body: HandoffCompleteRequest,
):
    await _resolve_handoff(board_id, issue_id, handoff_id)
    try:
        return await _svc.complete(
            handoff_id=handoff_id, actor=body.actor, payload=body.payload
        )
    except PayloadValidationError as exc:
        raise HTTPException(
            status_code=422,
            detail={
                "message": f"Validation failed for lane '{exc.lane}'",
                "lane": exc.lane,
                "errors": exc.errors,
            },
        )
    except (ValueError, ScopeDeniedError) as exc:
        raise HTTPException(status_code=422, detail=str(exc))
```

**Response shape on validation failure:**

```json
{
  "detail": {
    "message": "Validation failed for lane 'qa'",
    "lane": "qa",
    "errors": [
      {
        "loc": ["coverage_pct"],
        "msg": "Input should be less than or equal to 100",
        "type": "less_than_equal"
      },
      {
        "loc": ["test_results"],
        "msg": "Field required",
        "type": "missing"
      }
    ]
  }
}
```

The existing 422 path for `ValueError` (status check, legacy missing-field
detection) is unchanged: it still produces a string `detail`. This avoids
breaking any client that reads `detail` as a string.

## 6. Frontend — Minimal Error Parsing

`HandoffSection.vue::submitCompletion` is updated to read the structured
422 and surface per-field errors. The form layout and field inputs are
**not** changed. The error string is joined on a single line (the existing
`<p>` tag does not honor newlines without a `whitespace-pre-line` class —
which would be a template change).

```ts
async function submitCompletion() {
  if (!issue.value || !completingHandoffId.value || !completionPreview.value) return
  isCompleting.value = true
  completionError.value = null
  try {
    const payload: Record<string, unknown> = {}
    for (const field of completionPreview.value.missingFields) {
      const v = completionValues.value[field]?.trim()
      if (v) payload[field] = v
    }
    await boardStore.completeHandoff(
      issue.value.id, completingHandoffId.value, payload, 'user'
    )
    cancelCompletion()
  } catch (err: any) {
    // Structured 422 from P1.5 typed payload validation
    const detail = err?.data?.detail
    if (detail && typeof detail === 'object' && Array.isArray(detail.errors)) {
      const items = detail.errors.map(
        (e: any) => `${e.loc.join('.')}: ${e.msg}`
      )
      completionError.value = `${detail.message} — ${items.join('; ')}`
    } else {
      completionError.value = err?.data?.detail || err?.message || 'Failed to complete handoff'
    }
  } finally {
    isCompleting.value = false
  }
}
```

This is **one parsing change**, not a UI refactor. The form keeps its
current shape; the error display gets richer content.

## 7. Testing

### 7.1 New file — `backend/tests/test_payloads.py`

For each of the 8 lanes, run:

| Case | Assertion |
|---|---|
| Valid payload | `model_validate(...)` returns instance |
| Missing required field | `ValidationError`, `loc == [field]`, `type == "missing"` |
| Wrong type (e.g. `coverage_pct: "99"`) | `ValidationError`, appropriate `type` |
| Literal mismatch (`decision: "yes"`) | `ValidationError`, `type == "literal_error"` |
| `min_length` violation (`summary: ""`) | `ValidationError`, `type == "string_too_short"` |
| `max_length` violation (`summary: "x"*2001`) | `ValidationError`, `type == "string_too_long"` |
| Whitespace-only string (`"   "`) | `ValidationError` (stripped to empty, then `min_length=1` fails) |
| `ge`/`le` violation (`coverage_pct: 150`) | `ValidationError` |
| **Extra field** (`{"diff_summary": "x", "sneaky": 1}`) | `ValidationError`, `type == "extra_forbidden"` |
| Empty list (`acceptance_criteria: []`) when `min_length=1` | `ValidationError` |

Estimated ~50-60 cases.

## 9. Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Existing handoffs' payloads don't conform to new schema | Medium | Legacy handoffs can't complete | Legacy payloads are not retroactively validated. UI form pre-fills `missingFields` from preview, so the user can still supply missing fields. |
| `extra="forbid"` rejects legitimate legacy keys not in schema | Low | Completing some handoffs fails | All 8 lane schemas cover every `required_completion_fields` entry. If a real legacy key surfaces, it can be added to the schema in a follow-up. |
| Pydantic 2 error message format changes across versions | Low | E2E regression test could flake | Test asserts on `loc` and status code only, not on `msg` string. |
| Frontend error parsing change affects other 422 paths | Low | Other endpoints' 422 display changes | Parsing logic is local to `submitCompletion`. Falls back to existing string path when `detail` is not a structured object. |
| SQLite JSON round-trip loses list/dict types | Low | Read-back payload is wrong type | `model_dump(mode="json")` writes JSON-safe values. Existing list/dict round-trips are stable. |

## 10. Acceptance Criteria

- [ ] `PayloadValidationError` triggers a 422 with `detail.lane` and
      `detail.errors[]` populated.
- [ ] `check_payload(final_payload)` runs before Pydantic validation
      (regression test in `test_handoff_service.py`).
- [ ] Existing `ValueError` 422 path (status checks, legacy missing-field
      detection) is unchanged: returns string `detail`.
- [ ] Existing 200 happy path (create → accept → dispatch → complete) is
      unchanged for compliant payloads.
- [ ] All 8 lanes' `required_completion_fields` derived from `LANE_PAYLOADS`
      (single source of truth, no duplication).
- [ ] E2E `handoff-completion.spec.ts` is green, including the new
      structured-422 regression.
- [ ] Frontend `submitCompletion` displays per-field validation errors when
      the backend returns structured 422.
