### 1. Dev stack 全面 unblock（20:57，1 個 commit）

- `9d1e3bf` fix(dev): unblock backend startup + login

昨天 Plan J 4 個 dev 帳號 (`superadmin / admin@default / ops@default / user@default`) 在 commit log 上是上線了，但實際上 backend container **重啟不停**、登入 **500**。今天把 4 條故障一次拔掉：

1. **`email-validator==2.2.0` 補進 requirements.txt**
   J-3 tenant schemas 用 `pydantic.EmailStr`，但 image 沒裝這個套件 → import 階段就 raise → uvicorn 沒起來 → docker restart loop。

2. **`DELETE /tenants/{id}/members/{uid}` 拿掉 `status_code=204`**
   FastAPI 拒絕 204 + body 組合，response builder 內部 assertion 觸發 → 500。改回 200 + JSON body。

3. **`/auth/me` 補上 `Depends` 傳 user dict**
   J-3 codemod 在 `auth.py` 留下一個沒被 bound 的 `current_user` 自由變數，每個 `/auth/me` 呼叫都 500。改成把 resolved user dict 用 `Depends` 傳進 `get_current_user_info`。

4. **`tenant_scope_middleware` 從 BaseHTTPMiddleware 改寫成純 ASGI3**
   這條是今天最痛的一條 — `BaseHTTPMiddleware` 為每個 request 起一個 anyio task group，內層 ASGI app 跑在**不同的 event loop**，而 asyncpg 連線池綁在 lifespan 那條 loop。`session.execute()` 直接 raise `RuntimeError: Future attached to a different loop`。改成 plain ASGI callable 後整個 request 都跑在同一條 loop，cross-loop error 消失。

> 順手把 alembic_version.version_num 的 `varchar(32)` 太窄問題點出來（DDL 完成但 stamp 寫不下 39 char 的 0022 revision id），但當時還沒修 — 是 5740bfb 才上 0023。

### 2. Alembic 0023 拓寬 version_num（22:45，1 個 commit）

- `5740bfb` fix(alembic): 0023 widen alembic_version.version_num to varchar(64)

把昨天 9d1e3bf 點到的 trap 收掉。autogen 出來的 `alembic_version.version_num varchar(32)` 塞不下 0022 那個 39 字元的 revision id：

- Postgres 走 DDL autocommits，0022 的 `drop global unique + create composite unique` 已經落庫
- 但 stamp `INSERT INTO alembic_version ... '0022_per_tenant_unique_constraints'` 撞 `value too long`
- alembic 包在 migration transaction 內，stamp 失敗 → 整筆 rollback
- 下一次 `alembic upgrade head` 又看到 `current=0021`，再跑 0022，再炸，**60s 死循環**

當下 operator workaround 是手動 `ALTER COLUMN version_num TYPE varchar(64)` 把 live DB 救回來。0023 把同樣的 ALTER 烤進 migration chain，讓 fresh DB 不會再撞。Downgrade 故意是 no-op（縮回 varchar(32) 等於自找麻煩）。整段 ALTER 在 Postgres 是 metadata-only，不會 rewrite table。

> 提一下當下第一版有用 `information_schema` / `PRAGMA table_info` 判斷「已經寬就 no-op」 — 那條 introspection 在 dev asyncpg pool 裡 hang 住，最後用 plain `ALTER COLUMN` 收掉，理論上同 type ALTER 的開銷可以吞。

### 3. Plan J bug C-1：alembic 死結 → entrypoint 子行程（23:49，1 個 commit）

- `d998e5a` fix(j-bug-1): move alembic upgrade to entrypoint to break asyncio.to_thread deadlock

J-dev-handoff 報告點名 C-1 類 bug：即使 0023 stamp 過了，FastAPI lifespan 還是有可能 hang。根因是：

```python
await asyncio.to_thread(_run_alembic_upgrade_head)
```

`alembic.command.upgrade` 內部會呼叫 `asyncio.run(run_async_migrations())`，在 worker thread 開**新**的 event loop；那條 loop 結束時 dispose 引擎，dispose handshake 又跟外層 uvicorn loop 的 executor 互鎖 → lifespan 卡死。

修法乾淨俐落：**把 alembic 從 lifespan 移出來，丟到 docker entrypoint**：

```dockerfile
CMD ["sh", "-c", "cd backend && alembic upgrade head && exec uvicorn main:app ..."]
```

- 子行程有自己的 event loop，跨 loop 競態消失
- `exec uvicorn` 讓 uvicorn 變 PID 1，docker stop signal 正常傳遞
- alembic 失敗 → 容器重啟 + 印出錯訊（不再是「靜默起一台壞掉的 server」）
- lifespan 加 `SKIP_LIFESPAN_ALEMBIC=1` 環境變數檢查，docker 路徑直接 no-op
- 本機 pytest 沒設 env var，走 in-process branch 不受影響

### 5. Plan I 護航：nginx SSE-friendly location（16:05，1 個 commit）

- `7d893eb` chore(nginx): add /api/ai-studio/ SSE-friendly location

AI Studio 的對話介面是 token-by-token SSE 串流。沒顯式 `proxy_buffering off` 的話 nginx 會把回應緩起來，直到 upstream 關閉（或 read timeout）才 flush — chat UX 整個被打壞。

新增的 `/api/ai-studio/` location 鏡像 Plan F 的 WS 策略：

```nginx
location /api/ai-studio/ {
    proxy_pass http://backend:8000/api/ai-studio/;
    proxy_http_version 1.1;
    proxy_set_header Host $host;
    proxy_set_header Connection '';
    proxy_buffering off;
    proxy_cache off;
    proxy_read_timeout 3600s;
    proxy_send_timeout 3600s;
    add_header X-Accel-Buffering no always;
}
```

- `proxy_buffering off` + `X-Accel-Buffering: no` → 不緩衝
- `proxy_cache off` → 不共用 cache chunk（不同使用者不能混）
- `proxy_read/send_timeout 3600s` → 跟 WS 同級，長 pause 不會被切
- `Connection ''` → SSE 不用 Upgrade（不像 WS），但清掉 Connection header 避免 stale connection reuse

放在 catch-all `/` location **前面**，所以 `/api/ai-studio/*` 優先權高於 frontend proxy。既有 `/api/` location 留下來管其他 endpoint（health / RBAC / webhooks），SSE block 故意做窄一點，不影響其他 path 的預設行為。
