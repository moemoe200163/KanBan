### Plan F 設計重點（給未來讀者）

**Server 端 — 廣播層獨立於既有 `RunLogManager`**
- 新 `IssueConnectionManager`（`backend/api/v1/endpoints/ws.py`）：`Dict[issue_id, Set[WebSocket]]` + 反向 `Dict[WebSocket, issue_id]` 方便 `disconnect` 清理
- 公開 helper `broadcast_issue_updated(issue_id, change_type)`，發送的 payload 刻意保持極小（`{type, issue_id, change}`），client 自己知道怎麼 refetch —— **不在 wire 上送新狀態**，避免 writer / reader 結構漂移
- 三個寫入點（cycle_reports / ecc / issue_collaboration）都包 `try/except`，WS 壞掉不會讓 user-visible 流程失敗

**Client 端 — 兩個 composable，兩個獨立 WS**
- `useBoardLive` 掛在 `KanbanBoard.vue`：watch `columns`，自動訂閱/退訂 `issue_id` 集合；收到 nudge 全板 refetch
- `useIssueDetailLive` 掛在 `IssueDetail.vue`：訂一個 issue，回呼 `onUpdate(change)` 讓呼叫端決定 refetch 範圍
- 都刻意不做 reconnect、不做樂觀更新 —— 寫在 file header 註解裡，避免 scope creep

**nginx 配合**
- `/ws/` read/send timeout 拉到 `3600s`（預設 60s 會把 idle live channel 砍掉）
- `listen 80 default_server` 讓 devflow-nginx 成 host 預設 server（解 port-80 bare origin CORS）
- `location /` 移到最下面（讓 `/api/`、`/ws/`、`/deliveries/` 優先 match）

| 因素 | 細節 |
|------|------|
| **測試覆蓋** | 三個 broadcast hook 都沒有對應的 pytest 項目；只靠人工 `curl` 過 `/ws/issues` 訂閱/廣播 round-trip |
| **WS auth 放行** | Plan F 走的是 dev mode 開的匿名 WS（DEV_AUTH_BYPASS 連帶）；production 是否要 gate 待決定 |

| **scope creep guard** | 兩個 composable header 都寫了「no reconnect / no optimistic / no cross-issue」，不過這是 code comment、不是測試保證 |
